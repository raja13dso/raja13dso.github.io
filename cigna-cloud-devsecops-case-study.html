01

THE PROBLEM
The multi-cloud governance gap

Evernorth operates at the intersection of healthcare services, pharmacy benefits, and enterprise workforce management — a regulated environment where cloud adoption has accelerated faster than the governance structures designed to manage it safely. By the time I stepped into the BISO role, the organisation's cloud footprint had grown into a heterogeneous multi-cloud estate spanning AWS as the primary public cloud platform, Microsoft Azure for M365 and enterprise integrations, Oracle OCI for specialised workloads and legacy platform modernisation, and a sprawling catalogue of SaaS and PaaS vendors with varying levels of security assurance.

The fundamental governance problem was not that cloud was insecure by design — it was that no single, unified security framework governed how all three CSPs, plus the SaaS/PaaS layer, were assessed, configured, monitored, and remediated. Each platform had been adopted with its own security controls, its own configuration baselines, and its own monitoring approach. Gaps lived in the white space between them.

For a healthcare organisation subject to HIPAA, PCI-DSS, and Cigna's own rigorous internal security standards, that fragmentation represented material risk — not theoretical risk. The question was not whether there were misconfigurations across this estate. The question was how many, where, and which ones could cause a breach or compliance failure before they were found and fixed.

Multi-cloud is not a technology strategy. It is a risk distribution problem — and without unified governance, every cloud platform you add multiplies the surface area your adversaries can probe while your teams are looking elsewhere.

Framing principle — cloud security governance programme
Cloud Footprint Complexity

AWS, Azure, and OCI each present distinct IAM models, networking constructs, logging frameworks, and service catalogues. Governing them under a single coherent security standard requires abstraction — translating platform-specific controls into unified policy language that applies consistently across all three.

Cloud Deployed Faster Than Governance

Development teams were provisioning cloud resources through Infrastructure as Code at a pace that manual security review could not match. New S3 buckets, Azure storage accounts, OCI compute instances, and third-party SaaS integrations were appearing in the estate faster than they could be assessed against security baselines.

Each layer presented distinct security challenges. The SaaS layer presented a different problem entirely: vendor-managed infrastructure where Evernorth's control was limited to configuration of the application layer and access management — not the underlying platform.

03

RESEARCH PHASE
Mapping the real posture

The research phase for cloud security governance is a structured interrogation of the estate — not a checklist exercise. The goal was to understand not just what misconfigurations existed, but why they existed: what process gaps, tooling limitations, or governance absences allowed them to persist. Fixing symptoms without understanding causes produces a hamster wheel of remediation rather than durable security improvement.

Research Workstreams
STREAM A
Cloud Asset Inventory & Classification

Conducted a full inventory of cloud resources across AWS, Azure, and OCI — using CSPM tooling to enumerate accounts, subscriptions, and tenancies. Each resource was classified by data sensitivity (PHI-adjacent, PII, legal-privileged, or general) to establish a risk-weighted remediation priority model. Shadow cloud accounts — provisioned outside IT governance — were surfaced and brought into scope.

STREAM B
Configuration Baseline Benchmarking

Assessed current cloud configurations against CIS Benchmarks for AWS, Azure, and OCI — the industry-standard configuration baseline for each platform. Findings were triaged by severity and mapped to their regulatory consequence: HIPAA §164.312 technical safeguards, PCI-DSS v4.0 requirements 1 through 12, and Cigna's internal cloud security standards. This produced a remediation backlog with regulatory weight attached to each finding.

STREAM C
IAM Architecture Review

Reviewed identity and access management across all three CSPs — evaluating privilege levels, cross-account trust relationships, service account permissions, federation configurations, and MFA enforcement coverage. Particular attention was paid to over-privileged IAM roles in AWS and Azure AD conditional access policy gaps — two of the most commonly exploited misconfigurations in enterprise cloud environments.

STREAM D
DevSecOps Pipeline Maturity Assessment

Evaluated the security integration maturity of existing CI/CD pipelines — assessing whether IaC was being scanned for misconfigurations before deployment (shift-left), whether container images were scanned for vulnerabilities, whether secrets were managed through vault services rather than hardcoded, and whether security gates existed at the pipeline level to block high-risk deployments automatically.

STREAM E
SaaS & Vendor Security Assessment

Reviewed third-party SaaS vendors against a vendor security assessment framework — evaluating SOC 2 Type II reports, data processing agreements, HIPAA Business Associate Agreements (BAAs), encryption standards, incident notification SLAs, and data residency commitments. Identified vendors lacking adequate contractual security obligations for the sensitivity of data they processed.

The research confirmed the hypothesis: the most critical cloud risks were not exotic — they were classic misconfigurations that had accumulated over time because no automated, continuous detection mechanism existed to catch them before they became persistent findings.

STRATEGIC THINKING
The governance architecture

The strategic challenge in multi-cloud governance is building a programme that is platform-agnostic at the policy layer but platform-native at the control layer. A single security policy — "all storage containing PHI must be encrypted at rest with customer-managed keys" — must translate into three different technical implementations: AWS KMS on S3, Azure Key Vault on Blob Storage, and OCI Vault on Object Storage. The policy is universal; the control is specific.

My governance architecture was designed around five pillars, each addressing a distinct dimension of the multi-cloud risk problem.

01
Unified Policy Layer

A single cloud security policy framework — mapped to CIS, NIST CSF 2.0, and CSA CCM — that governs all CSPs, SaaS, and PaaS environments through translated platform-native controls.

02
Shift-Left IaC Security

Security guardrails embedded at the Infrastructure as Code layer — catching misconfigurations in Terraform and CloudFormation templates before any resource is ever provisioned to the cloud.

03
Continuous Cloud Posture

CSPM tooling providing continuous, automated detection of configuration drift across all cloud accounts — replacing point-in-time audit exercises with always-on monitoring.

04
Zero Trust IAM

Least-privilege IAM across all platforms — eliminating standing privileges, enforcing MFA, managing cross-account access through role assumption rather than long-lived credentials.

05
Vendor Risk Governance

Structured SaaS/PaaS vendor assessment framework — ensuring third-party cloud services meet security and compliance requirements before data is entrusted to them.

06
Risk-Linked Reporting

Cloud posture KRIs linked directly to regulatory consequence — giving business leaders a risk-weighted view of cloud security posture, not just a list of technical findings.

The Zero Trust principle was especially important in the multi-cloud context. Traditional perimeter-based security assumed that resources inside the corporate network were inherently trusted. In a multi-cloud environment with SaaS integrations, remote workforce access, and API-based service-to-service communication spanning three CSPs, there is no perimeter to defend. Zero Trust — verify explicitly, use least privilege, assume breach — provided the architectural philosophy that governed IAM design decisions across all platforms.

CHALLENGES &
Solutions paired

Multi-cloud governance at enterprise scale is where well-designed strategies encounter organisational, technical, and operational friction. Each challenge below represented a real impediment to the programme — and each required a solution tailored to the specific constraints of the Cigna environment.

⚡ Challenge 01
No Unified Visibility Across CSPs

AWS, Azure, and OCI each had siloed security tooling — AWS Security Hub, Microsoft Defender for Cloud, and OCI Security Zones operated independently with no consolidated view. Security teams could not answer "what is our overall cloud posture?" without pulling three separate reports and manually correlating findings across platforms and accounts.

✓ Solution Applied
CSPM-Driven Unified Posture Dashboard

Implemented a Cloud Security Posture Management (CSPM) layer providing a normalised, cross-platform view of misconfigurations, policy violations, and compliance posture across all three CSPs. Findings were deduplicated, severity-normalised, and mapped to regulatory controls — producing a single source of truth for cloud security posture that both security teams and business leadership could consume.

⚡ Challenge 02
IaC Deployed Without Security Review

Development teams were deploying Terraform modules and CloudFormation stacks at high velocity. Security review was a manual, post-deployment activity — meaning misconfigured resources were often already running in production before any security assessment occurred. Some misconfigurations had existed undetected for months.

✓ Solution Applied
Shift-Left IaC Security Gates

Integrated IaC security scanning (Checkov / Terraform Sentinel policies) directly into CI/CD pipelines — scanning every Terraform and CloudFormation change at the pull request stage before any deployment was approved. High-severity findings (open storage buckets, unrestricted security groups, disabled encryption) automatically blocked the pipeline. Medium findings generated advisory alerts without blocking. This moved security review from post-deployment remediation to pre-deployment prevention.

⚡ Challenge 03
Over-Privileged IAM Across All Platforms

IAM privilege creep is endemic in enterprise cloud environments that have grown organically over time. Across AWS, Azure AD, and OCI IAM, audit findings revealed service accounts with Administrator-equivalent permissions being used for routine automation tasks, human accounts with standing access to sensitive data stores, and cross-account trust relationships that were broader than the specific integrations they were established to support.

✓ Solution Applied
Least-Privilege IAM Remediation Programme

Led a structured IAM remediation programme across all three platforms — using AWS IAM Access Analyzer, Azure AD Access Reviews, and OCI IAM analysis to identify over-privileged principals. Implemented a just-in-time (JIT) access model for privileged operations, replacing standing admin access with time-bounded role assumption. Service accounts were scoped to the minimum permissions required for their specific function. MFA enforcement was extended to cover all human identities accessing cloud consoles and sensitive APIs.

⚡ Challenge 04
SaaS Vendor Blind Spots

A significant portion of HR and Legal IT's operational data — employee records, legal correspondence, benefits data — was processed through SaaS platforms with varying security maturity. Some vendors lacked current SOC 2 Type II reports. Several did not have signed HIPAA BAAs despite processing health-adjacent employee data. Data residency commitments were absent or ambiguous in legacy contracts.

✓ Solution Applied
SaaS Vendor Security Assessment Framework

Designed and implemented a tiered SaaS vendor security assessment process — categorising vendors by the sensitivity of data they processed and applying corresponding assessment rigour. Tier 1 vendors (processing PHI-adjacent or legally-privileged data) required current SOC 2 Type II, signed BAA, documented incident notification within 72 hours, and data residency confirmation. Vendors failing Tier 1 requirements were flagged for contract renegotiation or service migration. A vendor security register was established to track assessment status and renewal dates continuously.

⚡ Challenge 05
Container & Kubernetes Security Gaps

Containerised workloads running on AWS EKS and Azure AKS were deployed with base images that had not been scanned for known vulnerabilities, Kubernetes RBAC configurations that were broader than necessary, and network policies absent — meaning pods could communicate laterally within the cluster without restriction.

✓ Solution Applied
Container Security Programme

Implemented a container security programme spanning the full image lifecycle: base image scanning in the CI/CD pipeline (blocking deployment of images with critical CVEs), a curated approved base image registry, Kubernetes RBAC hardening aligned to least-privilege principles, network policy implementation to enforce pod-to-pod communication restrictions, and runtime security monitoring for anomalous container behaviour. OCI Kubernetes Engine (OKE) workloads received equivalent controls adapted to OCI's native security tooling.

⚡ Challenge 06
Secrets Sprawl Across Cloud Environments

Developer convenience and historical practice had resulted in secrets — API keys, database credentials, cloud service account keys — appearing in code repositories, CI/CD pipeline environment variables, and application configuration files stored in cloud object storage. Secret scanning had not been consistently enforced across all repositories and pipelines.

✓ Solution Applied
Secrets Management Centralisation

Mandated migration of all cloud service secrets to platform-native vault services — AWS Secrets Manager, Azure Key Vault, and OCI Vault — with rotation policies enforced programmatically. Integrated secret scanning into CI/CD pipelines to detect and block any commit containing credentials, API keys, or private certificates. A remediation sprint addressed the existing secrets backlog across repositories, with compromised credentials rotated as the highest priority. Developer guidance and training on secrets management best practices was embedded into the DevSecOps onboarding programme.

EXECUTION
How the programme was built

Execution of a multi-cloud governance programme at enterprise scale requires a phased approach — attempting to address every platform, every control domain, and every finding simultaneously is a path to programme paralysis. The programme was structured in three phases, each building on the foundations established in the previous phase.

Execution Phases
PHASE 1
Foundation — Visibility & Critical Remediation

Deployed CSPM tooling across all cloud accounts to establish unified posture visibility. Prioritised and remediated critical misconfigurations with direct regulatory exposure — open storage buckets, publicly accessible databases, disabled CloudTrail logging, MFA gaps on privileged accounts. Established the cloud security KRI dashboard providing real-time posture metrics to both security and business leadership. Duration: first 60 days.

PHASE 2
Prevention — Shift-Left DevSecOps Integration

Integrated IaC security scanning and container image scanning into CI/CD pipelines across HR IT and Legal IT development teams. Implemented secrets management centralisation and secret scanning enforcement. Deployed Kubernetes RBAC hardening and network policy controls across EKS and AKS clusters. Launched IAM least-privilege remediation programme across all three CSPs. Duration: days 60–120.

PHASE 3
Maturation — Governance, Vendor Risk & Continuous Improvement

Completed SaaS vendor assessment programme, negotiating contract updates for Tier 1 vendors lacking required security commitments. Established cloud security governance cadence — monthly posture reviews, quarterly cloud risk assessments, and architecture review gates for new cloud service adoptions. Developed cloud security champion capability within development teams. Duration: days 120+ (ongoing).

Pipeline Security Controls — What Was Built

The CI/CD security integration covered four enforcement layers: (1) IaC static analysis blocking high-severity misconfigurations before provisioning, (2) container image scanning blocking images with critical CVEs from promotion to production, (3) secret scanning preventing credential commits from merging into the main branch, and (4) dependency scanning alerting on open-source components with known exploitable vulnerabilities. Each control was tuned to minimise false positives that would erode developer trust in the programme.

A critical design decision was building security controls that developers experienced as helpful, not obstructive. Every automated pipeline gate included actionable remediation guidance — not just a failure notification, but a description of the specific misconfiguration, its risk rationale, and the exact change required to resolve it. This design reduced the friction of security enforcement and prevented the "just bypass the gate" culture that emerges when developers feel security controls are arbitrary.

MEASURABLE
Impact delivered

Cloud security governance programmes produce two categories of outcome: the risks prevented before they materialise (which are inherently harder to measure but represent the programme's core value) and the measurable improvements in posture, velocity, and compliance coverage that demonstrate the programme is working. Both categories are represented below.

Unified
Multi-Cloud Posture View

AWS, Azure, and OCI security posture consolidated into a single CSPM-driven dashboard — replacing three separate, manually correlated audit exercises with continuous, automated visibility.

Shift-Left
IaC Security Enforcement

IaC misconfigurations now caught at the pull request stage — before any cloud resource is provisioned — rather than discovered post-deployment through reactive scanning or incident response.

Zero
Standing Privileged Access

Just-in-time access model established for privileged cloud operations across all three CSPs — eliminating standing admin accounts and reducing the blast radius of any compromised identity.

100%
Secrets in Vault Services

All cloud service credentials migrated to AWS Secrets Manager, Azure Key Vault, or OCI Vault — eliminating hardcoded secrets in code repositories, pipeline variables, and configuration files.

Tier 1
SaaS Vendors Assessed

All SaaS vendors processing PHI-adjacent or legally-privileged data formally assessed against security requirements — BAAs in place, SOC 2 reports current, data residency documented.

↓ MTTR
Cloud Finding Remediation

Reduction in mean time to remediate cloud security findings — driven by automated detection, business-weighted prioritisation, and developer-friendly remediation guidance embedded in pipeline alerts.

The measure of a cloud security programme is not the number of findings it surfaces — any scanner can produce a finding list. The measure is how quickly the organisation moves from finding to fixed, and whether the same misconfiguration category stops appearing over time. That requires cultural change as much as technical control.

Raja Kumar Sharma — on cloud security programme outcomes

The DevSecOps integration delivered an impact that compound over time: by catching misconfigurations at the IaC layer, the programme prevented a class of cloud security findings from ever reaching the production environment. The posture improvement is not visible in a "findings resolved" count — it is visible in the findings that no longer appear in production scans because they were blocked at the source. This prevention-oriented metric is the most meaningful indicator of a mature DevSecOps programme.

REFLECTION
What multi-cloud governance teaches

Multi-cloud security governance in a regulated healthcare enterprise is one of the most technically and organisationally complex security challenges in the industry today. It sits at the intersection of cloud architecture, identity engineering, developer tooling, regulatory compliance, and stakeholder communication — and it demands competence across all of these simultaneously.

Four reflections stand out from this programme that carry broader lessons for cloud security leadership.

Reflection 01
Governance Without Visibility Is Just Policy

A cloud security policy that cannot be verified continuously is not a security control — it is a document. The foundational investment of the programme was establishing automated, continuous posture monitoring across all three CSPs before attempting any other governance activity. Without that visibility, every subsequent discussion about cloud security risk was speculative. With it, discussions became evidence-based and actionable.

Reflection 02
Developer Trust is a Security Asset — Protect It

Every security gate added to a CI/CD pipeline is a potential point of friction that can erode developer goodwill toward the security programme. The design principle of providing actionable, specific remediation guidance with every pipeline failure — rather than a generic "security check failed" error — was not a courtesy. It was a strategic investment in the developer relationship that makes the entire DevSecOps programme sustainable over time.

Reflection 03
IAM is the Most Important Control in Cloud Security

Across all three CSPs, the highest-risk findings consistently traced back to identity and access management: over-privileged roles, missing MFA, broad trust relationships, long-lived credentials. Cloud infrastructure misconfigurations can expose data — but a compromised privileged identity can exfiltrate it, modify it, and cover its tracks. IAM hardening consistently produces the highest security return on investment of any cloud control domain.

Reflection 04
Shared Responsibility Requires Active Management

The shared responsibility model — CSPs secure the cloud infrastructure; customers secure what runs on it — is well understood in theory and routinely misapplied in practice. Healthcare organisations sometimes assume that choosing a HIPAA-eligible CSP service transfers security responsibility. It does not. Actively managing the customer's side of the shared responsibility boundary, across every service and every CSP, is the unglamorous operational core of enterprise cloud security governance.

The multi-cloud estate is not a technical problem to be solved once. It is a living system that grows, changes, and accumulates risk continuously. The security programme that governs it must be equally dynamic — continuously monitoring, continuously adapting, and continuously educating the engineering culture that shapes the estate every day.

Raja Kumar Sharma — on sustaining cloud security governance at enterprise scale
